# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Product

**DaFit Canvas** ‚Äî a cross-platform desktop app for visually designing smartwatch faces compatible with the **Da Fit / MoYoung** ecosystem. Output of the editor is a `.zip` package that the external `dawft` CLI converts into a binary watch face uploaded via `dawfu`.

Watch hardware target:
- **240√ó240** screen, **16-bit RGB565**, little-endian, no alpha
- Type **C** binary (`fileType C`, `fileID 0x81`, `faceNumber 50001`)

## Current state vs target stack

Stack in tree today: React 19 + TS 6 + Vite 8, **Zustand** store, **JSZip** for packaging, **axios** for the MoYoung API, **react-router-dom** for routing, **lucide-react** for icons. Editor canvas uses plain HTML5 Canvas (no Konva yet). No Tauri shell yet ‚Äî runs as a web app via `npm run dev`.

Source layout under `src/`:
- `components/` ‚Äî UI components; editor-specific UI lives in `components/editor/`.
- `lib/` ‚Äî domain logic (`dawft.ts` for Type C codec; `faceN.ts` for FaceN; `projectIO.ts` orchestrates import/export, materialization, insert/rebind helpers; `bmp.ts`, `moyoung.ts` API client).
- `store/editorStore.ts` ‚Äî single Zustand store for the editor (project, selection, error, all actions).
- `types/` ‚Äî shared TS types (`face.ts` for project model, `moyoung.ts` for API).
- `pages/` ‚Äî top-level routes (`Editor`, `WatchFaces` browser, `Pack` test tool).

Constraints to keep in mind: fast startup, low memory, **offline-first**, pixel-perfect rendering.

The downloaded watch-face corpus in `faces-corpus/` (gitignored) is the source of truth for "what the firmware actually accepts" ‚Äî refer to it before changing TYPE_TABLE, sharing rules, or default dimensions. Regenerate stats with `node --experimental-strip-types scripts/analyze-corpus.ts ./faces-corpus`.

## TypeScript gotchas

`tsconfig.app.json` sets `verbatimModuleSyntax: true`, `noUnusedLocals`, `noUnusedParameters`, `erasableSyntaxOnly`. Practical effects:
- Use `import type { Foo }` for type-only imports ‚Äî plain `import { Foo }` will fail the build when `Foo` is a type
- No enums or constructor-parameter properties (`erasableSyntaxOnly`) ‚Äî use unions/const objects instead
- Unused locals/params are errors, not warnings

`tsconfig.json` is a solution file; edit `tsconfig.app.json` for `src/` and `tsconfig.node.json` for build-tool configs.

## Domain: watch face format

Knowledge below is product-specific and not derivable from code ‚Äî keep it in mind when working on `export/`, `canvas/`, or element definitions.

### `watchface.txt` structure

Header (required):
```
fileType    C
fileID      0x81
dataCount   <count of faceData lines>
blobCount   <total BMPs>
faceNumber  50001
```

Then one `faceData` line per element:
```
faceData   <TYPE>   <INDEX>   <X>   <Y>   <W>   <H>   # comment
```
- `TYPE` ‚Äî hex element type (table below)
- `INDEX` ‚Äî starting blob index in the BMP sequence
- `X, Y` ‚Äî top-left position
- `W, H` ‚Äî width / height

A `blobCompression` section is auto-generated by `dawft` ‚Äî **the editor does not emit it**.

### Element TYPE ‚Üí blob count

Each element type consumes a fixed number of `dbNNN.bmp` files starting at `INDEX`. **Source of truth is [TYPE_TABLE in src/lib/dawft.ts](src/lib/dawft.ts) ‚Äî update there, not here, when adding a type.** The table below highlights the categories.

- **Single-blob static**: `0x01` BACKGROUND (full 240√ó240), `0xce/0xd0/0xd1/0xda` BATT_IMG_*, `0x45/0x46` TIME_AM/PM, `0x71/0x81/0x91/0xa1` *_LOGO, `0xa5/0xa6` DIST_KM/MI, `0xc0/0xc1` BTLINK_UP/DOWN, `0xf0` SEPERATOR, `0xf1‚Äì0xf5` HAND_*.
- **Multi-blob animated**: `0x00` BACKGROUNDS (10 strips of 240√ó24, not a full background), `0xf6` TAP_TO_CHANGE, `0xf7/0xf8` ANIMATION (count = `animationFrames` from header).
- **Multi-blob digit sets (count = 10 each)**: `0x30` DAY_NUM, `0x40/0x41/0x43/0x44` TIME_H1/H2/M1/M2, `0x6b/0x6c` MONTH_NUM_B/DAY_NUM_B, `0x72‚Äì0x74` STEPS_B/B_CA/B_RA, `0x82‚Äì0x84` HR_B/B_CA/B_RA, `0x92‚Äì0x94` KCAL_B/B_CA/B_RA, `0xd2‚Äì0xd4` BATT/_CA/_RA, plus less common: `0x11` MONTH_NUM, `0x12` YEAR, `0x62‚Äì0x68` STEPS/HR/KCAL variants, `0xa2‚Äì0xa4` DIST/_CA/_RA, `0xd7‚Äì0xd9` WEATHER_TEMP variants.
- **Multi-blob name strings**: `0x60` DAY_NAME (7), `0x61` DAY_NAME_CN (7), `0x10` MONTH_NAME (12).
- **Multi-blob progress bars (count = 11)**: `0x70` STEPS_PROGBAR, `0x80` HR_PROGBAR, `0x90` KCAL_PROGBAR, `0xa0` DIST_PROGBAR.

`blobCount` in the header must equal the sum of blobs across all `faceData` entries (less when entries share an `idx` ‚Äî see Sharing below).

### Sharing rules (faceData entries that point at the same `idx`)

The dawft format allows two or more `faceData` entries to share a blob range by reusing the same `idx`. 64% of real faces use sharing somewhere. **But which type-combos can share is firmware-policed**, not free-form ‚Äî sharing an unsupported combo renders garbage on the watch.

The exhaustive list of "seen in the wild" share combos lives in [faces-corpus/_type-defaults.json](faces-corpus/_type-defaults.json) (`safeShareList`). Highlights:

- **Always safe**: `0x40+0x41+0x43+0x44` (all 4 TIME digits), `0x40+0x41` (TIME hours pair), `0x43+0x44` (TIME minutes pair), `0x11+0x30` (MONTH+DAY digits), `0x6b+0x6c` (their B-variants).
- **Documented**: `0x73+0x83`, `0x73+0x93`, `0x83+0x93` and supersets (B_CA digit sets across HR/STEPS/KCAL).
- **NEVER safe**: `0x70+0x90` (STEPS_PROGBAR + KCAL_PROGBAR) and any progress-bar pair ‚Äî zero faces in the corpus do this. The rebind action in `projectIO.ts` should refuse these.

When in doubt, regenerate `_type-defaults.json` from the corpus and consult the `safeShareList`.

### File identification

- `fileID` is `0x81` in 84% of faces. `0x84` (14%) and `0x04` (2%) also appear ‚Äî likely different watch model variants. Don't hardcode 0x81 in import/decode paths; preserve whatever the file declares.
- `faceNumber` is opaque to the firmware (just an ID for the user's catalogue) ‚Äî any 16-bit value is legal.
- `animationFrames` is 0 in 76% of faces; non-zero values populate the `0xf6/0xf7/0xf8` animation slots and are stored at `sizes[0]` in the binary header.

### BMP output

- Windows BMP, **16-bit RGB565, little-endian, no alpha**
- Filename pattern: `db000.bmp`, `db001.bmp`, ‚Ä¶ (zero-padded, sequential, matches `INDEX`)
- RGB565 encoding:
  ```ts
  const r = (pixel.r >> 3) & 0x1F
  const g = (pixel.g >> 2) & 0x3F
  const b = (pixel.b >> 3) & 0x1F
  const rgb565 = (r << 11) | (g << 5) | b
  ```

### Dynamic placeholders (preview-only in MVP)

Tokens that render as live values in the editor preview but are emitted as digit/icon blobs at export time: `{{HH}}`, `{{mm}}`, `{{DD}}`, `{{MM}}`, `{{BATTERY}}`, `{{STEPS}}`, `{{HEART_RATE}}`.

### Export pipeline

1. Render each element to its frame(s) on an offscreen 240√ó240 canvas
2. Encode each frame as RGB565 BMP ‚Üí `dbNNN.bmp`
3. Emit `watchface.txt` with header + one `faceData` per element
4. Zip everything ‚Üí user runs `dawft create folder=‚Ä¶ output.bin` then `dawfu upload address=XX:XX:XX:XX:XX:XX output.bin`

## Engineering rules

- Strong TypeScript typing ‚Äî no `any` in domain types (Element, Layer, Project)
- Modular by feature folder (`canvas/`, `export/`, etc.), not by file kind
- Reusable hooks for canvas interaction (drag, resize, select) and store access
- Avoid premature abstraction ‚Äî three similar lines beats a generic helper
